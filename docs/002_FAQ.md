## Purpose

이 문서는 LectureOS의 방향을 이해할 때 반복해서 생기는 질문에 직접 답한다. Manifesto와 Product의 본문을 복제하지 않으며, 제품 경계와 확정된 판단의 이유, 현재 가설, 미결정 사항을 구분한다.

## Decision Status

- **Confirmed**: 현재 L0 Blueprint에서 승인된 방향
- **Working Assumption**: 현재 작업을 위해 사용하지만 영구 결정은 아닌 가설
- **Deferred Decision**: 후속 제품 또는 설계 단계에서 결정
- **Requires Validation**: 자료나 실제 사용을 통한 검증이 필요한 항목
- **Out of Scope**: 현재 제품 경계 밖

L0의 `Immutable`은 절대 변경 금지가 아니라, 기술 변화만으로 쉽게 흔들리지 않는 상위 수준의 기준이라는 뜻이다. FAQ는 반복 질문과 검증 결과가 쌓이면 Manifesto보다 자주 갱신될 수 있다.

## Product Identity and Boundaries

### Q1. LectureOS는 무엇인가?

**Status: Confirmed**

한국어 장시간 교육 강의를 촬영한 뒤 발생하는 반복적인 후반작업과 검수를 줄이는 제품이다. 현재 제품 계약과 경계는 [LectureOS Product](001_PRODUCT.md)를 따른다.

### Q2. 왜 단순한 자막 생성기가 아닌가?

**Status: Confirmed**

실제 병목은 음성을 텍스트로 바꾸는 한 단계가 아니다. 교정, 자막 분할과 타이밍, 불확실한 구간 확인, 편집 판단, 시간축 연결이 같은 검수 과정에서 반복되기 때문이다.

이 답이 LectureOS가 모든 후반작업을 직접 수행한다는 뜻은 아니다. 제품의 In Scope와 Out of Scope는 [LectureOS Product](001_PRODUCT.md#in-scope)에 정의한다.

현재 흐름과 목표 흐름은 다음과 같이 구분한다.

~~~text
현재: 원본 영상 → 초안 SRT → Final Cut에서 컷편집과 자막 수정 동시 수행
목표: 원본 영상 → LectureOS 분석·교정·라벨링·검수 → 승인된 자막과 편집 결정 → Final Cut에서 화면 편집과 최종 확인
~~~

### Q3. 가장 먼저 최적화하는 사용자는 누구인가?

**Status: Confirmed**

장시간 한국어 교육 강의를 직접 후반작업하는 교육자 또는 소규모 교육 콘텐츠 운영자다. 모든 언어와 영상 장르를 동시에 지원하는 범용 플랫폼으로 시작하지 않는다.

### Q4. LectureOS는 전문 영상 편집기를 대체하는가?

**Status: Out of Scope**

아니다. 화면 구성, 색보정, 합성, 복잡한 컷 편집, 최종 렌더링 같은 전문 시각 편집 기능 전체는 현재 범위 밖이다. LectureOS는 분석, 텍스트, 시간축, 검수 정보와 사용자 결정을 연결하는 데 집중한다.

### Q5. Final Cut은 LectureOS의 영구적인 전제인가?

**Status: Working Assumption**

아니다. Final Cut은 현재 핵심 사용자의 실제 작업 환경이며 초기 검증에 중요한 맥락이다. 특정 편집 도구에 대한 장기 의존이나 통합 범위는 확정하지 않았다.

### Q6. LectureOS는 LMS, 녹화, 스트리밍 또는 OCR 제품인가?

**Status: Out of Scope**

아니다. 수강생 관리, 촬영과 송출, 콘텐츠 배포, 범용 OCR은 현재 제품 경계 밖이다. 자세한 범위는 [LectureOS Product](001_PRODUCT.md#out-of-scope)를 따른다.

### Q7. LectureOS가 ASR이나 LLM 모델 자체를 만드는가?

**Status: Out of Scope**

아니다. ASR과 LLM은 제품이 활용할 수 있는 교체 가능한 구성요소다. LectureOS의 책임은 특정 모델을 소유하는 데 있지 않고, 그 결과를 근거와 함께 검수 가능한 작업으로 연결하는 데 있다.

### Q8. 지금부터 모든 언어와 교육 형식을 고려해야 하는가?

**Status: Confirmed**

확장 가능성을 불필요하게 막지는 않지만 현재 핵심 사용 사례를 먼저 최적화한다. 범용화를 이유로 한국어 장시간 교육 강의의 실질적인 검수 문제를 뒤로 미루지 않는다. 관련 판단 규칙은 [Principle 14](004_PRINCIPLES.md#principle-14-optimize-the-current-core-use-case-first)를 따른다.

## Data and Meaning

### Q9. LectureOS에는 하나의 Source of Truth가 있는가?

**Status: Confirmed**

하나의 파일이나 Transcript만으로 진실을 단순화하지 않는다. 다음 계층은 서로 다른 책임과 근거 수준을 가진다.

1. **원본 미디어**: 촬영된 영상과 오디오 자체이며 물리적 사실의 최상위 근거다.
2. **원본 시간축 기반 발화·단어 데이터**: 발화나 단어가 원본 미디어의 어느 시점에 존재하는지 나타내는 시스템 기준 데이터다.
3. **Transcript**: 원본 발화를 사람이 읽고 교정할 수 있도록 표현한 의미 계층이다.
4. **Subtitle**: 학생이 읽기 좋도록 분할·정리된 파생 결과다.
5. **Timeline**: 원본 또는 편집 후 시간축을 표현하는 파생 구조다.
6. **Review Item**: 사람의 확인이 필요한 오류·변경·편집 후보다.

각 계층의 정확한 데이터 구조는 향후 Glossary와 Data Model에서 확정한다.

### Q10. 왜 Transcript 하나를 절대적인 Source of Truth라고 하지 않는가?

**Status: Confirmed**

Transcript는 사람이 의미를 확인하고 교정하는 계층이므로 원본 발화, 인식 결과, 교정 결과가 달라질 수 있다. 단일 최종 텍스트만 남기면 실제 오디오 위치와 변경 과정을 추적할 수 없다.

### Q11. raw 인식문, 교정된 Transcript, Subtitle은 왜 분리하는가?

**Status: Confirmed**

각 표현의 목적이 다르기 때문이다. raw 인식문은 모델이 낸 결과를 보존하고, 교정된 Transcript는 발화 의미를 사람이 확인하며, Subtitle은 학생의 가독성을 위해 분할하고 정리한다. 한 계층의 변경이 다른 계층의 근거를 덮어쓰면 안 된다.

### Q12. “Word Timeline”이 확정된 용어인가?

**Status: Deferred Decision**

아니다. 현재는 **원본 시간축 기반 발화·단어 데이터**라는 임시 표준 표현을 사용한다. 정확한 모델명과 `Word`, `Utterance`, `Segment`의 단위는 `010_GLOSSARY.md`와 `030_DATA_MODEL.md`에서 확정한다.

### Q13. SRT 같은 출력 파일은 기준 데이터인가?

**Status: Confirmed**

아니다. SRT는 Subtitle을 전달하기 위한 파생 출력 형식이다. 필요한 정보와 사용자의 결정이 보존되어 있다면 조건에 맞춰 다시 생성할 수 있어야 한다.

### Q14. Timeline은 영상 편집기 자체를 뜻하는가?

**Status: Confirmed**

아니다. 이 문서에서 Timeline은 원본 또는 편집 후 시간축을 표현하고 서로 연결하는 파생 구조를 뜻한다. 구체적인 편집 도구 UI나 구현 방식은 의미에 포함하지 않는다.

### Q15. Review와 Review Item은 같은 뜻인가?

**Status: Deferred Decision**

현재 임시 사용에서 Review는 사람이 확인하는 활동 또는 흐름을, Review Item은 확인이 필요한 개별 오류·변경·편집 후보를 뜻한다. Review는 읽기 전용 Report가 아니며 자막과 Edit Candidate에 대한 Accept, Reject, Modify와 결정 보존을 포함한다. 정확한 용어와 상태 모델은 Glossary와 Data Model에서 확정한다.

## Human and Automation

### Q16. “사람이 판단하고 AI가 추천한다”는 모든 자동 처리를 금지한다는 뜻인가?

**Status: Confirmed**

아니다. 근거가 명확하고 되돌릴 수 있으며 구조적으로 검증 가능한 저위험 작업은 자동화할 수 있다. 교육적 의미를 바꾸거나 원본을 훼손할 수 있는 결정은 더 보수적으로 다뤄야 한다. 기준은 [LectureOS Principles](004_PRINCIPLES.md)를 따른다.

### Q17. AI가 교정문이나 편집 후보를 제안할 수 있는가?

**Status: Confirmed**

가능하다. 다만 제안은 근거와 불확실성을 함께 가져야 하며, 시스템이 검증할 수 없는 의미 변화나 교육적 판단을 확정된 사실처럼 다루면 안 된다.

### Q18. LLM이 최종 타임스탬프를 정할 수 있는가?

**Status: Confirmed**

직접 통제하지 않는다. LLM은 텍스트, 의미 경계, 분류 또는 변경 후보를 제안할 수 있지만, 최종 시간 값과 순서·겹침 같은 구조적 유효성은 코드가 검증하고 결정해야 한다. 자세한 규칙은 [Principle 5](004_PRINCIPLES.md#principle-5-let-llms-propose-and-code-validate-structure)와 [Principle 6](004_PRINCIPLES.md#principle-6-keep-timestamps-out-of-llm-control)을 따른다.

### Q19. AI가 불필요하다고 판단한 구간을 자동으로 삭제하는가?

**Status: Confirmed**

현재 기본 방향은 설명 가능한 추천과 사람의 확인이다. 특히 교육적 가치가 불명확하면 보수적으로 유지한다. 자동화 수준 확대는 실제 오탐 비용을 검증한 뒤 별도 승인할 문제다.

### Q20. 반복 설명과 잡담은 항상 삭제 대상인가?

**Status: Confirmed**

아니다. 반복은 복습일 수 있고, 짧은 일화는 이해나 집중에 기여할 수 있다. 표면적인 중복이나 분류만으로 교육적 가치를 확정하지 않는다.

### Q21. 자동화의 성공은 정확도 숫자로 판단하는가?

**Status: Confirmed**

정확도는 중요한 진단 지표지만 최종 제품 목표는 총 검수 시간 감소다. 중요한 의미 오류를 숨기지 않으면서 사용자가 강의를 완성하는 시간이 실제로 줄어야 한다.

## Current Strategy and Future Options

### Q22. LectureOS는 local-first 제품인가?

**Status: Working Assumption**

현재 전략적 기본값은 local-first다. 장시간 원본의 크기, 개인정보, 반복 비용, 현재 사용자 환경을 고려한 선택이다. 다만 영구 불변 원칙이나 local-only 제품으로 확정하지 않으며 선택적 클라우드 기능을 막지 않는다.

### Q23. local-first의 장기 지위는 언제 결정하는가?

**Status: Requires Validation**

실제 처리 성능, 개인정보 요구, 협업 필요, 운영 비용을 검증한 뒤 제품 요구사항과 아키텍처 단계에서 결정한다. 현재 Principles에서는 `Working Principle`로만 다룬다.

### Q24. LectureOS는 오픈소스인가?

**Status: Deferred Decision**

아직 확정하지 않았다. 공개 가능성을 막지 않는 문서화와 공급자 독립성은 유용하지만, 라이선스와 공개 범위는 별도 제품 결정이 필요하다.

### Q25. LectureOS는 SaaS인가?

**Status: Deferred Decision**

아직 확정하지 않았다. 협업이나 선택적 클라우드 기능의 가능성은 열어두되, SaaS를 현재 사업 모델이나 제품 경계로 가정하지 않는다.

### Q26. 오픈소스 가능성과 SaaS 가능성을 함께 열어두는 것이 결정인가?

**Status: Confirmed**

현재 확정된 것은 어느 한쪽을 불필요하게 막지 않는다는 경계뿐이다. 구체적인 공개, 운영, 수익 모델은 제품 책임자의 후속 결정이다.

### Q27. 특정 ASR, LLM, alignment 모델을 기본값으로 정했는가?

**Status: Requires Validation**

아니다. 실제 한국어 장시간 강의 자료에서 품질, 검수 비용, 처리 제약을 비교해야 한다. L0에서는 공급자와 모델 독립성만 판단 기준으로 둔다.

### Q28. Edit Candidate와 자동 컷편집은 같은가?

**Status: Confirmed**

아니다. 강의 구간을 분석하고 컷편집용 라벨과 Edit Candidate를 만들어 사람이 검수하도록 하는 흐름은 현재 핵심 제품 범위다.

Edit Candidate를 사람의 확인 없이 자동 적용하는 컷편집, FCPXML 생성, 특정 편집 도구와의 직접 통합은 별도의 Deferred Decision이다. 전문 편집기 전체 대체와 사람 확인 없는 교육적 자동 삭제는 현재 Out of Scope다.

### Q29. V1에는 정확히 무엇이 들어가는가?

**Status: Deferred Decision**

L0 문서에서 단계별 기능 목록을 확정하지 않는다. [LectureOS Product](001_PRODUCT.md)가 제품 경계를 정의하고, 구체적인 V1 수용 범위는 향후 Product Requirements에서 결정한다.

## Blueprint Governance

### Q30. FAQ가 L0이면 수정할 수 없는가?

**Status: Confirmed**

수정할 수 있다. L0는 구현 세부보다 오래 유지되는 기준이라는 뜻이다. 반복 질문, 검증 결과, 승인된 제품 결정이 바뀌면 상태, 날짜, Change Log와 함께 갱신한다.

### Q31. FAQ와 다른 문서가 충돌하면 무엇을 따르는가?

**Status: Confirmed**

질문의 주 소유 문서를 따른다.

- 존재 이유: `000_MANIFESTO.md`
- 제품 정의와 경계: `001_PRODUCT.md`
- 장기 목적지: `003_VISION.md`
- 제품·시스템 판단 규칙: `004_PRINCIPLES.md`
- 반복 질문과 상태 설명: 이 FAQ

후속 승인 문서가 생기면 그 문서가 소유한 세부 범위 안에서 최신 승인 결정을 확인한다.

### Q32. 코드나 현재 도구가 L0 문서와 다르면 무엇을 따르는가?

**Status: Confirmed**

차이를 먼저 확인하고 제품 책임자의 최신 결정을 따른다. 현재 구현이 있다는 이유만으로 제품 철학을 바꾸지 않고, 오래된 문서라는 이유만으로 코드 변경을 자동 승인하지도 않는다.
